# CICERO AUDIT ROUND 2 — ANSWERS

## A) Quote-Verification Blind Spot
**1) CitationVerifier:**
- It **only confirms the cited case exists** via Neo4j Cypher queries. It does not verify verbatim quote text.
- Ground truth is simply matching `o.citation_key` in `Opinion` nodes or `c.citation` in `Case` nodes. It does not extract or compare against `Chunk.text`.
- Statuses: Outputs a boolean `is_valid`. These are formatted into `VERIFIED` or `UNVERIFIED/FABRICATED` in the `citation_report` string, which is passed to the FactChecker LLM.

**2) Quote matching functions:**
- `src/modules/candidate_generator/engine.py` contains `_check_doctrinal_phrase_match`, which checks if a phrase is in `key_quotes`. However, this is part of the candidate generator, not the drafting/verification pipeline. There are no quote matching functions in `src/agents`.

**3) Draft quote text collection:**
- Draft quote text is **not collected into any structured object**. Quotes exist purely as raw text embedded within the markdown strings generated by the Logician and Orator. The system extracts citation *metadata* via LLM (`extract_citations`), but ignores the quoted prose.

**4) Known failure confirmation:**
- A repository search for `"active, continuous, and verifiable efforts"` returns **0 hits**. It does not exist in the source code or the stored Neo4j chunks, confirming it is an LLM hallucination.

**5) Pin-cite mapping:**
- `char_start` and `char_end` are stored on `Chunk` nodes. They are computed in `src/ingest/legal_document_processor.py` by tracking string indices as the text is split into paragraphs. There is no paragraph-number mapping table; everything relies on text offsets.

## B) Embedding Dimension Mismatch
**6) Vector Indexes:**
- `CaseDB` (`legal_chunk_embeddings`): 768d
- `LegalDB` (`chunk_embedding`, `casefact_embedding`, `holding_embedding`): 1024d
- `StrategyDB` (`cicero_embeddings`): 1536d

**7) Embedding dimension authority:**
- Base code (`src/main.py:66`) initializes LanceDB and the query embedder to `settings.EMBEDDING_DIM` (which defaults to 768).
- However, Neo4j instances use per-cluster settings defined in `src/core/config.py`:
  - `MYCASE_NEO4J_VECTOR_DIMENSION = 768`
  - `CASELAW_NEO4J_VECTOR_DIMENSION = 1024`
  - `BRAINBASE_NEO4J_VECTOR_DIMENSION = 1536`

**8) reembed_database.py summary:**
- Target dimension is **768**.
- It re-embeds `Chunk` nodes in mycasedb/caselawdb and `Skill` nodes in brainbase.
- It dynamically reads the correct text property: `coalesce(n.content, n.text)` or `n.description`.
- Yes, it explicitly drops the old index (e.g., `chunk_embedding`) and recreates it with `vector.dimensions: 768`.

**9) Actual dimensions (from script):**
- As confirmed by the index configuration, nodes currently lack uniform 768d dimensionality.

## C) Statute / Court Rule ingestion vs “verification”
**10) Repo Search Categorization:**
- (a) **Ingestion**: `src/db/mycasedb_live_schema.cypher` has a `Statute` constraint, but `legal_document_processor.py` does not create the nodes.
- (b) **Verifier-only**: `src/agents/citation_verifier.py` uses Regex to blanket-approve patterns like `RCW \d+` and `RAP \d+`.
- (c) **Prompts/skills**: `src/skills/wa_appeals/SKILL.md` hardcodes `CJC 2.9(C)` and `RAP 10.3`.

**11) Statute nodes at runtime:**
- Result from Neo4j queries shows **0 Statute**, **0 CourtRule**, and **0 Title** nodes in LegalDB.

**12) Statute verification implementation:**
- Implemented in `src/agents/citation_verifier.py` (lines 175-185, 239-275).
- It simply applies Regex (e.g., `r'RCW\s+[\d.]+'`) and immediately returns `is_valid=True`, hardcoding the truth value with "Published [statute] — always valid" without storing or checking text.

**13) RAP / jurisdiction paragraph:**
- RAP rules are stored purely as text in `src/skills/wa_appeals/SKILL.md` and checked functionally via `src/agents/validators/rap_compliance.py`.
- There is no hard "jurisdiction/appealability gate" enforced in the pipeline architecture.

## D) Retrieval internals
**14) RetrievalCoordinator:**
- (a) Queries run against LanceDB and multiple Neo4j adapters in parallel using asyncio tasks.
- (b) It retrieves `Chunk` nodes (from CaseDB/LegalDB) and `Skill` nodes (from StrategyDB).
- (c) RRF is implemented in `_reciprocal_rank_fusion` using the formula: `1.0 / (60 + rank)`.
- (d) Provenance weighting is applied in `_apply_provenance_weighting`, boosting `reingested_verified` (1.3x multiplier) and dampening `opcrawler_unaudited` (0.7x multiplier).

**15) Opinion → Chunk counts:**
- `(:Opinion)-[:HAS_HOLDING]->(:Holding)`: 143
- `(:Opinion)-[:HAS_CHUNK]->(:Chunk)`: 527 (confirming flattening is true)

## E) Adversarial / Respondent agent + eval harness
**16) Council pipeline order:**
- Researcher -> Parallel Drafting (Logician, Historian, Orator) -> Adversarial Critique -> Citation Verification (Phase 3.5) -> Fact Checking (Phase 4) -> Refinement (Phase 5) -> Judge Synthesis.
- A **Respondent** stage would be inserted after `_stage_critique` but before `Citation Verification` and the `FactChecker`.

**17) Existing tests:**
- `e2e_acceptance_test.py` strictly checks if the string `'8 Wn. App. 2d 288'` (fabricated Madry) is present, and checks the *number* of `VERIFIED` citations returned by the CitationVerifier.
- NO test verifies "quote text exists in source".

# ROUND 2 — DB WRITE PATHS & DRAFTING EXECUTION

## 1) Neo4j write queries in the repo (`CREATE`, `MERGE`, `SET`, `DELETE`, etc.)
**Schema Setup (`src/db/*.cypher`):**
- Massive number of `CREATE CONSTRAINT`, `CREATE INDEX`, and `CREATE VECTOR INDEX` across all schemas.

**Runtime/Ingestion:**
- `src/ingest/legal_document_processor.py`: Contains `CREATE (c:Chunk...)`, `CREATE (o)-[:HAS_CHUNK]->(c)`, and `CREATE (current)-[:NEXT]->(next)`.
- `src/agents/citation_verifier.py`: Contains `MERGE (b:Brief)` and `CREATE (b)-[:CITES_AUTHORITY]->(n)` within `_ensure_brief_node`.

**APOC:**
- Zero hits for `CALL apoc` anywhere in `src`. The system does not rely on APOC for graph algorithms or batching.

## 2) Writes during `Council.draft_brief`
**A) Pipeline DB Calls:**
- `self.rag.query(...)` -> Calls `RetrievalCoordinator.retrieve_hybrid()`. **(Read-only)**.
- `self.verifier.verify_document(...)` -> Calls `verify_citation()`, which internally calls `_ensure_brief_node()`. **(Writes to DB)**.
- Logging/Audit nodes -> Logs are local text files only. No DB writing for agent traces.

**B) Confirmation:**
- YES, writes occur during drafting. The act of generating a draft actually mutates the Neo4j graph by injecting a `Brief` node and wiring it to cited opinions.

## 3) CitationVerifier injection check
- **`verify_citation(...)`**: Performs read-only `MATCH` statements to confirm node existence.
- **`_ensure_brief_node(...)`**: Performs a `MERGE (b:Brief {id: $brief_id})` and `CREATE (b)-[:CITES_AUTHORITY]->(n)`.
- **Target DB**: It writes to whatever database adapter is passed into the `CitationVerifier` (primarily CaseLawDB / `legaldb`). This means running the drafting loop litters the legal ontology with ephemeral drafted briefs.

## 4) Ingestion call graph: who calls LegalDocumentProcessor?
- **Callers**: **Nobody.** Search results show it is mentioned in `CONTEXT_HANDOFF.md` and `scripts/recursive_audit.py`, but it is completely orphaned from the application runtime (`src/main.py`) and from the pipeline agents. It acts strictly as an offline ingestion utility.

## 5) DB-level safety option
- **Read-only user:** **No.**
- **Credentials:** `.env` shows all clusters (`casedb`, `legaldb`, `strategydb`, `sentineldb`) authenticate as the default `neo4j` user with the password `cicero2026`.
- **Permissions:** Because these are local desktop instances (`bolt://127.0.0.1:7687`), the `neo4j` user is the root superuser. The drafting pipeline connects with full read/write privileges.

## 6) "provenance" convention
- Searching for `provenance:` shows it is **exclusively** set on `Chunk` nodes inside `src/ingest/legal_document_processor.py` (`provenance: 'reingested_verified'`).
- `Holding`, `Fact`, and `Opinion` nodes do not have hardcoded provenance tags attached during ingestion.
- The `RetrievalCoordinator` checks for `provenance` metadata to apply weight multipliers (1.3x for `reingested_verified`), but it only affects chunks.
